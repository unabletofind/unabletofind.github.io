Detection Rules

The following rules are written for Microsoft Sentinel (KQL) and MDE Advanced Hunting. Tune thresholds to your environment's baseline before deploying as alerts.

Rule 1 — [Detection Name e.g. Suspicious PowerShell Spawned from Office Process]
Sentinel · Analytics Rule 
KQL
// Detects [what this rule catches — be specific]
// Severity: High | Tactic: Execution | TID: T1059.001

DeviceProcessEvents
| where Timestamp > ago(1d)
| where InitiatingProcessFileName in~ ("WINWORD.EXE", "EXCEL.EXE", "OUTLOOK.EXE")
| where FileName == "powershell.exe"
| where ProcessCommandLine has_any ("-enc", "-nop", "-w hidden", "iex", "DownloadString")
| project Timestamp, DeviceName, AccountName,
         InitiatingProcessFileName, ProcessCommandLine,
         SHA256
| order by Timestamp desc
Rule 2 — [Detection Name e.g. Anomalous DNS Beaconing to Known C2 Domains]
MDE · Advanced Hunting 
KQL
// Detects DNS queries matching known C2 patterns
// Severity: High | Tactic: C2 | TID: T1071.004

let c2_domains = dynamic([
  "[malicious-domain1.com]",
  "[malicious-domain2.net]"
]);
DeviceNetworkEvents
| where Timestamp > ago(7d)
| where RemoteUrl has_any (c2_domains)
   or RemoteIP in ("[C2 IP 1]", "[C2 IP 2]")
| summarize
    ConnectionCount = count(),
    FirstSeen = min(Timestamp),
    LastSeen  = max(Timestamp)
  by DeviceName, RemoteUrl, RemoteIP
| where ConnectionCount > 3
| order by ConnectionCount desc
Rule 3 — IOC Hash Hunt
Sentinel · Hunting Query 
KQL
// Hunt for known-malicious file hashes across device telemetry

let malicious_hashes = dynamic([
  "[SHA256_hash_1]",
  "[SHA256_hash_2]"
]);
DeviceFileEvents
| where Timestamp > ago(30d)
| where SHA256 in (malicious_hashes)
| project Timestamp, DeviceName, AccountName,
         FileName, FolderPath, SHA256, InitiatingProcessFileName
| order by Timestamp desc
Tuning Notes Before deploying Rule 1 as an alert: baseline your environment for legitimate [process spawning patterns]. Exclude [specific legitimate processes / accounts / paths] to reduce false positives. Start with Rule 3 (IOC hunt) as it has zero false positive risk.

IR Playbook

This playbook covers the response steps if you receive an alert matching this attack pattern. Steps are sequenced by priority — contain first, investigate in parallel, recover last.

Containment
  1. Contain
    [Action — e.g. Isolate the Affected Device]
    In MDE → Device page → Isolate device. This cuts all network connectivity except MDE management plane. Confirm isolation status shows "Isolated" in device timeline. Document isolation time in ticket.
  2. Contain
    [Action — e.g. Disable Compromised Account / Revoke Sessions]
    In Entra ID → User → Disable account + Revoke all sessions. If MFA was bypassed (AiTM indicator), also rotate [relevant credential type]. Block sign-in via Conditional Access if identity is confirmed compromised.
Investigation
  1. Investigate
    [Action — e.g. Pull Device Timeline — T-48h Window]
    In MDE device timeline, filter to [48h before first alert]. Look for: parent process anomalies, unusual child processes, file drops in [TEMP / AppData / ProgramData], registry modifications, and outbound connections to new IPs/domains. Export timeline for evidence.
  2. Investigate
    [Action — e.g. Scope Lateral Movement]
    Run KQL Rule 2 above across all devices. Check for any authentication events from the compromised account to other systems in the [same 48h window]. Flag any devices that had contact with identified C2 IPs/domains.
  3. Investigate
    [Action — e.g. Validate IOCs Against Org Environment]
    Run the IOC hash hunt (Rule 3) with a 30-day lookback. Pull all unique devices that touched flagged IOCs. This defines your incident scope for the report.
Eradication & Recovery
  1. Recover
    [Action — e.g. Wipe & Reimage or Live Response Remediation]
    If persistence confirmed: wipe and reimage. If no persistence found and malware didn't execute: use MDE Live Response to delete [specific files/registry keys], then run a full AV scan. Confirm clean before removing isolation.
  2. Recover
    [Action — e.g. Block IOCs Organisation-Wide]
    Add confirmed C2 IPs/domains to [Sentinel watchlist / MDE indicators / Forcepoint DLP block list]. Set expiry to 90 days, schedule review. Document all blocks added in ticket with justification.
